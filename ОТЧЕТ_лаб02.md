## Теоретическая часть

### Изученные концепции

- **Логическая структура:** Иерархия: Кластер БД → Базы данных → Схемы → Объекты (таблицы, представления)

- **Физическая структура:** Табличные пространства связывают логические объекты с физическим расположением файлов на диске

- **Системный каталог:** Системные таблицы (pg_class, pg_database, pg_namespace) и представления (pg_tables, pg_views) хранят метаданные

- **TOAST (The Oversized-Attribute Storage Technique):** Механизм хранения больших данных вне строки с различными стратегиями хранения

### Ключевые термины

- **Табличное пространство:** Логическая единица, связывающая объекты БД с каталогом на диске

- **Системный каталог:** Набор системных таблиц и представлений, содержащих метаданные о структуре БД

- **TOAST:** Технология автоматического выноса больших значений полей за пределы основной таблицы

- **Search_path:** Параметр, определяющий порядок поиска схем при обращении к объектам

- **Нежурналируемая таблица:** Таблица, операции с которой не записываются в WAL-журнал

#### Задача 2: Исследование pg_tables

**Цель:** Изучить представление pg_tables и понять разницу между таблицей и представлением

**Выполненные действия:**

```sql
\d+ pg_tables
```

**Результаты:**

```bash
# Описание представления pg_tables
                          View "pg_catalog.pg_tables"
   Column    |  Type   | Collation | Nullable | Default | Storage | Description 
-------------+---------+-----------+----------+---------+---------+-------------
 schemaname  | name    |           |          |         | plain   | 
 tablename   | name    |           |          |         | plain   | 
 tableowner  | name    |           |          |         | plain   | 
 tablespace  | name    |           |          |         | plain   | 
 hasindexes  | boolean |           |          |         | plain   | 
 hasrules    | boolean |           |          |         | plain   | 
 hastriggers | boolean |           |          |         | plain   | 
 rowsecurity | boolean |           |          |         | plain   | 
View definition:
 SELECT n.nspname AS schemaname,
    c.relname AS tablename,
    pg_get_userbyid(c.relowner) AS tableowner,
    t.spcname AS tablespace,
    c.relhasindex AS hasindexes,
    c.relhasrules AS hasrules,
    c.relhastriggers AS hastriggers,
    c.relrowsecurity AS rowsecurity
   FROM pg_class c
     LEFT JOIN pg_namespace n ON n.oid = c.relnamespace
     LEFT JOIN pg_tablespace t ON t.oid = c.reltablespace
  WHERE c.relkind = ANY (ARRAY['r'::"char", 'p'::"char"]);
```

**Выводы и объяснения:** pg_tables - это представление (view), а не таблица. Разница: таблица физически хранит данные на диске, представление - это виртуальная таблица, результат запроса к другим таблицам. Данное представление объединяет информацию из pg_class, pg_namespace и pg_tablespace для удобного доступа к метаданным о таблицах.

#### Задача 5: Анализ метакоманды

**Цель:** Изучить, какие запросы к системному каталогу выполняются при использовании метакоманды \d+ pg_views

**Выполненные действия:**

```sql
\d+ pg_views
```

**Результаты:**

```bash
                         View "pg_catalog.pg_views"
   Column   | Type | Collation | Nullable | Default | Storage  | Description 
------------+------+-----------+----------+---------+----------+-------------
 schemaname | name |           |          |         | plain    | 
 viewname   | name |           |          |         | plain    | 
 viewowner  | name |           |          |         | plain    | 
 definition | text |           |          |         | extended | 
View definition:
 SELECT n.nspname AS schemaname,
    c.relname AS viewname,
    pg_get_userbyid(c.relowner) AS viewowner,
    pg_get_viewdef(c.oid) AS definition
   FROM pg_class c
     LEFT JOIN pg_namespace n ON n.oid = c.relnamespace
  WHERE c.relkind = 'v'::"char";
```

**Выводы и объяснения:** Метакоманда \d+ pg_views показывает структуру представления pg_views. Это представление выполняет запрос к pg_class (для получения информации об объектах) и pg_namespace (для получения имён схем), фильтруя только представления (relkind = 'v'). Функция pg_get_viewdef возвращает текст определения представления.

## Анализ и выводы

### Основные наблюдения

1. Логическая и физическая организация: PostgreSQL четко разделяет логическую структуру (базы данных, схемы) и физическую (табличные пространства, файлы). Это позволяет гибко управлять размещением данных.

2. Системный каталог как источник истины: Все метаданные хранятся в системных таблицах, что обеспечивает согласованность и позволяет получать информацию о структуре БД средствами SQL.

3. TOAST - оптимизация хранения больших данных: Механизм TOAST автоматически оптимизирует хранение больших значений, обеспечивая баланс между производительностью и использованием дискового пространства.

### Сравнительный анализ

- Табличные пространства vs. файловая система: Табличные пространства предоставляют больший контроль над физическим размещением данных по сравнению с хранением всего в PGDATA.

- Журналируемые vs. нежурналируемые таблицы: Нежурналируемые таблицы обеспечивают более высокую производительность записи, но теряют данные при сбое.

- Различные стратегии хранения TOAST: Выбор стратегии (plain, extended, external, main) зависит от конкретного случая использования и характеристик данных.
